## Загальна інформація
- Джерело: корпус українських політичних виступів ParlaMint-UA та внутрішні дані університету
- Тип даних: текстові документи
- Формат: CSV/JSONL з токенами, лемами, POS, та витягнутими сутностями
- Мови: українська
- Кількість документів: ~N (точну кількість можна вписати після аналізу processed_v2.csv)
- Ліцензія/доступ: внутрішній, для навчальних цілей

## Лабораторні роботи

### ЛР2 — Cleaning & Normalization
- Вилучено шум (непотрібні символи, html-теги)
- Стандартизовано розділові знаки та пробіли
- Маскування PII: `<PHONE>`, `<EMAIL>`, `<URL>`, `<ID>`

### ЛР3 — Lemma / POS
- Створено колонки:
  - `lemma_text` — текст з нормалізованими лемами
  - `upos_seq` — послідовність Universal POS тегів
- Використано Stanza для лематизації та тегування
- Проведено аналіз помилок: суржик, скорочення, власні назви

### ЛР4 — Rule-based IE
- Витягнуто мінімум 3 типи полів (DATE, AMOUNT, PHONE)
- Використано regex + словники (місяці, валюти, номери телефонів)
- Створено:
  - `lab4_gold_ie.jsonl` — gold subset для оцінки
  - `ie_policy.md` — опис правил витягу
  - `audit_summary_lab4.md` — precision по полях, аналіз помилок
- Edge cases: `ie_edge_cases.jsonl` (20+ тестів)
## ЛР5 Splits & leakage

- Датасет містить офіційний розподіл на train/dev/test.
- Одиницею split є sentence_id (речення).
- Було перевірено можливі дублікати та near-duplicates між split.
- Критичних ознак data leakage не виявлено.

## TF-IDF + Logistic baseline (Lab 6)

Для задачі класифікації частин мови (upos) було побудовано baseline-модель
на основі TF-IDF та Logistic Regression.

Використано два варіанти:
- Baseline 1: тільки token (unigram)
- Baseline 2: token + lemma (unigram + bigram)

Основні результати:
- Accuracy: до ~0.82
- Macro-F1: до ~0.69

Виявлені проблеми:
- модель часто переоцінює клас PUNCT
- короткі токени (1–2 символи) важко класифікуються
- присутній дисбаланс класів
- відсутній контекст (модель бачить лише окремий токен)

Основні ризики:
- overlap класів (наприклад, службові слова)
- noisy tokens (числа, назви брендів)
- дисбаланс даних
- відсутність контексту

Подальші кроки:
- використання char n-grams
- додавання контекстних ознак
- балансування класів
